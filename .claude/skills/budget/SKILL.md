# Budget agent (financial aggregator/analyst)

## What it is, and what it deliberately is NOT

`agents/budget_agent.py` sits ON TOP of every other agent — it has no vendor
of its own, no client_accounts platform, no connection card. It pulls
already-collected data from `admin_service` (our cost/margin),
`google_ads_agent`/`meta_ads_agent` (real ad spend), and each platform
agent's own usage function (`avatar_agent.get_monthly_usage`,
`media_agent.get_monthly_usage`, `seo_agent.get_connected_tool`,
`website_agent.is_provisioned_by_us`). It never re-implements any of their
data collection — if a number is missing, that's a gap in THIS agent's
visibility, not a reason to build a second path to fetch it.

## The one rule that matters more than any feature: confidence labels

Every figure this agent returns carries one of three labels, and the module
docstring explains them in detail:

- **`hard`** — a real number from our DB or a live vendor API call.
- **`estimate`** — arithmetic from a REAL usage figure × a published rate
  (e.g. HeyGen minutes × HeyGen's published $/min range). The usage is real;
  the rate is a reference, not this client's verified invoice.
- **`unknown`** — genuinely no visibility, and no honest way to estimate one
  (Higgsfield's $ cost, ElevenLabs usage). Say so — never guess.

**Never collapse these into one blended number.** If you're extending this
agent, a new figure without an honest confidence label is a bug, not a
detail to add later.

## What's real vs. estimated vs. unknown today (investigated 2026-07-20)

| Source | Status | Why |
|---|---|---|
| Our internal cost/margin | `hard` | Reused from `admin_service.get_client_admin` — unchanged, single source of truth. |
| Google/Meta ad spend | `hard` | Live API via `get_campaign_performance` — same data the ad agents themselves use. |
| HeyGen avatar minutes | `hard` | HeyGen's own render duration, tracked by `avatar_agent`. |
| HeyGen avatar **cost** | `estimate` | Minutes × a published $1-4/min range (varies by avatar model, which isn't recorded per video). HeyGen's `v2/user/remaining_quota` wallet endpoint exists but its semantics after the Feb-2026 PAYG migration weren't verifiable with confidence — NOT wired in; flagged for a future one-round check. |
| Higgsfield credits used | `hard`-ish | Real when their job response includes a `credits` field (unverified against a live key — see the media skill); no $ conversion exists. |
| Higgsfield **cost in ₪/$** | `unknown` | No spend/billing API, no published $-per-credit rate. Genuine gap — do not estimate this. |
| ElevenLabs usage/cost | `unknown` | No usage or billing endpoint queried at all. |
| SEO tool (SEOptimer/SEMrush/Ahrefs) subscription cost | `estimate` | Vendor's published list price for the entry tier (`SEO_TOOL_LIST_PRICE_USD_MONTH`, checked 2026-07-20) — not the client's actual invoice/currency/discount. |
| InstaWP hosting (Phase-2 sites) | `hard`, but **OUR cost, not the client's** | Known constant from `PRICING["website"]["new_site_hosting"]`; real but not yet written into `client_costs` — see internal_cost_gaps below. |

## Forecast — the client's own stated numbers

`get_forecast(client_id)` reads the newest matching `leads` row: the free-
text `marketing_budget`/`organic_budget` answers from onboarding, plus the
`market_reality`/`goals_90_days`/`kpis` already generated by
`onboarding_agent.build_proposal`.

**Known gap**: `leads` has no `client_id` column, so the match falls back to
"newest lead row with this client's email" (the same approximate join
`api_server.py`'s checkout already uses to backfill leads). The checkout
handler now ALSO tries to stamp `client_id` on that row in a try/except that
silently no-ops until the column exists —

**add a nullable `client_id bigint` column to `leads` in Supabase** to make
this exact. Until then, `forecast.source` always says which method matched
("client_id (exact)" vs. "email (approximate...)") so nothing is hidden.

Free-text budget parsing (`_extract_ils_amount`) is deliberately
conservative: it returns `None` rather than guess whenever the text has 0 or
3+ numbers in it. A missing forecast number is a normal, expected outcome —
never coerced to 0.

## Drift + alerts

`ad_spend_vs_forecast` compares real 30-day ad spend against the parsed
monthly budget — only computed when both sides are real/parseable. The
weekly scan (`run_weekly_scan`, `GET /api/budget/scan`, X-Admin-Key) raises
`agent_alert` on:

- Ad spend > `ad_spend_drift_pct_high`% (default 130) or < `_low`% (default
  50) of the stated budget.
- Any connected ad channel spending ≥ `zero_conversion_spend_floor_ils`
  (default 300₪) in 30 days with **zero** recorded conversions.

All three thresholds live in `app_settings` (same pattern as
`low_margin_threshold_pct`) — tune via `PUT /api/admin/settings`, no deploy
needed. Alerts dedup 6 days per `issue_key` (`details->>issue_key`, same
idiom as `website_agent`'s health scan) so an ongoing deviation nudges
weekly, not every run.

Deliberately NOT alerted on: anything with `estimate`/`unknown` confidence
(a HeyGen cost estimate is too fuzzy to trigger a real alert) and the
InstaWP hosting gap (informational, not a recurring deviation).

## Trend (real drift over time, not just a snapshot)

Each weekly scan records a `budget_snapshot_recorded` activity row (reuses
`client_activity`, no new table) — `get_trend(client_id)` reads these back
as a real time series. **Empty until the scan has actually run a few times
— no retroactive history exists** before this agent shipped; the function
says so rather than fabricating a backfilled series.

## Internal cost gaps (OUR side)

`get_internal_cost_gaps` flags real internal costs `cost_tracker` doesn't
record yet — v1 covers InstaWP hosting for Phase-2-provisioned sites (real
constant from PRICING, but nothing writes it into `client_costs`).
`get_financial_picture` folds it into `our_revenue_and_cost_adjusted`
(shown alongside, never replacing, the official `our_revenue_and_cost` from
admin_service) as a read-time patch. The real fix — `website_agent.
provision_site` recording a recurring cost row — is flagged, not built,
to avoid a second untested write path in the same change.

## Narrative (optional, on-demand, never invents a number)

`generate_narrative(client_id)` (`POST /api/admin/clients/{id}/budget/
narrative`, admin session) is the ONLY LLM call in this agent — everything
else above is deterministic. It receives the already-computed
`get_financial_picture` output and is instructed to explain it in Hebrew for
Johnny, carrying each figure's confidence into the sentence, and explicitly
forbidden from inventing a number not present in the input. NOT run
automatically by the weekly scan (would burn a Claude call per client per
week for a UI few people click) — fetched lazily from the drawer's "✨ ניתוח
קצר" button.

## Endpoints

Admin (session cookie): `GET /api/admin/clients/{id}/budget` (the full
picture — heavier, live ad-API calls), `GET /api/admin/clients/{id}/budget/
trend`, `POST /api/admin/clients/{id}/budget/narrative`.
Cron (X-Admin-Key): `GET /api/budget/scan` (weekly).
Client (session cookie): `GET /api/client/external-costs` now delegates to
`budget_agent.get_client_facing_costs` — same URL/shape as before, plus a new
`external_tools` section.

Scheduler (weekly, mirrors the seo-cycle pattern):

```
gcloud scheduler jobs create http budget-scan --schedule="0 7 * * 1" \
  --uri="{SERVICE_URL}/api/budget/scan" --http-method=GET --update-headers=X-Admin-Key={ADMIN_KEY}
```

## Deferred / flagged

- `leads.client_id` column (see above) — the biggest single accuracy
  upgrade available for the forecast side.
- HeyGen wallet-balance endpoint (`v2/user/remaining_quota`) — real signal,
  unverified semantics post-migration, not wired in.
- InstaWP hosting cost as a proper recurring `client_costs` write (currently
  a read-time patch only).
- No trend charting UI yet — `get_trend` is wired but the drawer doesn't
  render a sparkline from it (data will be too sparse to be useful until a
  few weekly scans have actually run).
